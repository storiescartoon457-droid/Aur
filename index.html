AI Pipeline

How a Photo Becomes a Diagnosis

End-to-end computer vision and language model pipeline — from raw image upload to personalized recommendations.

1
Image Intake & Validation
User uploads 1–3 photos (front face, left profile, hair/scalp close-up). The server validates: file format (JPEG/PNG/WebP), resolution (min 480×640), filesize limit (10 MB). Image hash is computed for deduplication. The original is never stored — only a processed version.
Sharp.js (resize/compress) EXIF strip (privacy) SHA-256 hash S3 pre-signed URL
2
Face Detection & Landmark Extraction
MediaPipe Face Mesh (468 landmarks) or dlib's 68-point model runs on the image. Detects bounding box, confirms a face is present and sufficiently lit. Landmarks are used to compute facial geometry — distance ratios between forehead width, cheekbone width, jawline width, and face height determine face shape category.
MediaPipe Face Mesh OpenCV Facial ratio algorithm Confidence threshold 0.85
3
Hair Segmentation
A semantic segmentation model (BiSeNet or MediaPipe Hair Segmentation) isolates the hair region from the background and face. This creates a binary mask used downstream for all hair analysis — color, density, frizz, and shine are all computed only within this mask to avoid false readings from background pixels.
BiSeNet / MediaPipe Semantic segmentation Binary mask IoU > 0.88
4
Skin Analysis (Vision Model)
The face region (minus eyes, lips, hair, beard) is passed to a fine-tuned EfficientNet-B4 or a custom CNN trained on dermatology datasets. It classifies skin type and detects concerns as a multi-label classification problem. Each concern (acne, pigmentation, dark circles, etc.) is output as a severity score 0–100. The under-eye region is cropped separately for dark circle analysis.
EfficientNet-B4 (fine-tuned) Multi-label classification Severity 0–100 per concern Dermnet / HAM10000 dataset Age regression head
5
Hair Analysis (Texture & Signal Processing)
Within the hair mask, multiple algorithms run in parallel: (a) FFT-based frequency analysis detects frizz level by measuring high-frequency noise in the texture; (b) Laplacian of Gaussian measures edge sharpness for split-end detection; (c) Variance of local pixel intensities indicates density; (d) Specular highlight detection using HSV channels estimates shine level; (e) Fiber orientation analysis classifies straight/wavy/curly/coily type.
FFT frizz detection LoG edge detection HSV specular analysis Fiber orientation (curl classifier) Watershed for density map
6
Score Computation
All numeric outputs are fed into a weighted scoring formula. Face score = weighted average of symmetry (25%), skin clarity (40%), grooming (20%), and age-appropriate feature presence (15%). Hair score = health (35%) + shine (20%) + density (20%) + frizz penalty (−15%) + split-end penalty (−10%). Both scores are normalized to 0–100 with pleasant rounding and capped at 99 (no perfect score).
Weighted formula Min-max normalization Score capped at 99
7
LLM Recommendation Generation
All extracted features and scores are structured as a JSON context object and sent to Claude Sonnet (or GPT-4o) via the API. The system prompt instructs the model to act as a certified dermatologist and trichologist, and the user prompt contains the full analysis JSON. The LLM outputs: explanation of score, strengths, improvement areas, a 7-day skincare routine, hair routine, product suggestions, lifestyle tips, and style recommendations — all personalized to the exact detected features.
Claude Sonnet / GPT-4o Structured JSON context RAG with product database Dermatologist system prompt Streaming response
8
Progress Comparison
If a previous scan exists in the database, the new feature vector is compared with the old one using cosine distance for the embedding, and delta values are computed for all numeric metrics. Visual diff is generated by aligning both face images using landmark-based affine transforms and computing per-region pixel difference maps. The AI Coach is then given the delta to narrate progress in natural language.
Feature vector delta Affine face alignment Pixel diff map LLM progress narrative
Technology Stack

Why Each Technology Was Chosen

Every choice is opinionated and justified — built for scale, privacy, and AI-first development.

FRONTEND

React Native + Expo

Single codebase for iOS and Android. Expo's managed workflow handles camera permissions, push notifications, and file system access out of the box. React Native Reanimated 3 enables the smooth spring animations the design calls for. Expo EAS handles CI/CD builds and OTA updates without app store resubmission for minor changes.
BACKEND

Node.js + Fastify + Python FastAPI (Microservices)

Fastify handles REST API routes (auth, user data, scans) at ~30k req/s. Python FastAPI serves the AI/ML pipeline — Python is the native language for OpenCV, MediaPipe, and TensorFlow. Two services communicate via internal HTTP. This avoids forcing Python to handle auth logic or the ML server to deal with database ORM patterns.
AI — VISION

MediaPipe + EfficientNet-B4 + ONNX Runtime

MediaPipe runs on-device for face detection (latency <50ms, no server round-trip for initial detection). The fine-tuned EfficientNet-B4 skin model is exported to ONNX format for consistent inference across Python and potential future on-device deployment. ONNX Runtime gives 2–4× faster inference than raw TensorFlow at equivalent accuracy.
AI — LLM

Claude Sonnet API (Anthropic) or GPT-4o

Used exclusively for natural language recommendation generation — not for vision tasks. Claude Sonnet provides nuanced, medically-grounded output with strong instruction-following for structured JSON. The LLM receives a structured JSON object (not the image) for privacy reasons — only extracted metrics, never raw photos, reach the LLM.
DATABASE

PostgreSQL (Supabase) + Redis + Pinecone

PostgreSQL stores all relational data (users, scans, routines, progress). Supabase provides managed Postgres with real-time subscriptions, row-level security, and built-in auth. Redis caches latest scan results and session data for <10ms response. Pinecone (vector DB) stores face embedding vectors for progress comparison and celebrity similarity matching.
CLOUD

AWS (S3 + Lambda + ECS Fargate + CloudFront)

S3 stores processed images with per-user AES-256 encryption. Lambda handles image preprocessing at scale (no idle compute cost). ECS Fargate runs the Python AI container without managing EC2 instances. CloudFront CDN serves the app's static assets and recommendation content globally with <50ms latency.
AUTH

Supabase Auth (Google, Apple, Email OTP)

Supabase Auth handles JWT issuance, refresh token rotation, and OAuth flows with Apple Sign-In (required for iOS App Store) and Google. Passwordless email OTP reduces friction for new users. Row-Level Security policies in Postgres ensure users can only query their own data — even if the API is bypassed.
NOTIFICATIONS

Expo Push Notifications + OneSignal

Expo Push handles token management for both iOS (APNs) and Android (FCM) from a single API. OneSignal is added for A/B testing notification copy, scheduling daily reminders, and segmenting users by behavior (e.g. users with hair concerns get different nudges than skin-focused users).
PAYMENTS

RevenueCat + Stripe

RevenueCat abstracts iOS App Store and Google Play subscriptions into a single SDK, handling receipt validation and entitlement management. Stripe handles web purchases and one-time PDF export fees. RevenueCat's dashboard provides conversion funnel analytics across both stores.

Privacy Architecture

Built Private by Default

Every design decision treats facial data as the most sensitive possible input. Privacy is not a compliance checkbox — it's a core feature users will trust the app with their face based on.

🔒

EXIF Stripping on Upload

All image metadata (GPS location, device model, timestamp) is stripped server-side before any processing begins. The raw file from the user's phone never reaches the database.

🧬

Only Metrics Reach the LLM

The AI recommendation engine (Claude/GPT-4o) never receives the photo. Only the extracted numeric scores and text labels are sent — treating the LLM as a recommendation engine, not a vision model.

🔑

Per-User Encryption (AES-256)

Images stored in S3 are encrypted with AES-256. Each user's photos are encrypted with a unique key derived from their user ID, so a breach of one key cannot decrypt another user's data.

🗑️

Photos Auto-Delete Option

Users can opt for "process only" mode — images are deleted from S3 within 60 seconds of analysis completion. Only the extracted metrics and scores are stored, not the photo.

🛡️

Row-Level Security (Supabase RLS)

PostgreSQL RLS policies ensure that every query, even if the application layer is bypassed, can only return data belonging to the authenticated user. Scans, recommendations, and conversations are all RLS-protected.

🌍

GDPR + CCPA Compliance

One-tap "Delete All My Data" permanently deletes the user's photos, analysis data, chat history, and account. Data export (JSON) satisfies GDPR Article 20 (portability). Retention policy: raw photos ≤ 90 days unless overridden by user preference.

🚫

No Third-Party Training

User photos and analysis data are never used to train third-party models or shared with advertisers. The privacy policy explicitly prohibits this and is surfaced during onboarding before any photo is taken.

📋

Consent at Every Step

Explicit opt-in consent is collected before the first scan (biometric data), before enabling progress comparison (storing multiple face photos), and before enabling the AI Coach (storing conversation history linked to scan context).

🔍

Audit Logging

All access to user photo records is logged with timestamp, requesting service, and action type. Anomalous access patterns (bulk queries, off-hours access) trigger automatic alerts to the security team.
